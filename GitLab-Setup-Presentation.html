на удалённом сервере через VS Code + Docker

Подключение по SSH → установка Docker → запуск GitLab в контейнере

2. Сгенерировать SSH-ключ и добавить в Selectel

2.1 Сгенерировать ключ в PowerShell

# генерируем пару SSH-ключей (приватный + публичный) по алгоритму ed25519
ssh-keygen -t ed25519

На все вопросы можно нажимать Enter (путь по умолчанию C:\Users\<User>\.ssh\id_ed25519, пароль — опционально).

В результате появятся два файла:

id_ed25519       — приватный ключ (никому не передавать)
id_ed25519.pub   — публичный ключ (его добавляем в Selectel)

2.2 Скопировать публичный ключ

# выводим публичный ключ и сразу копируем его в буфер обмена
Get-Content $env:USERPROFILE\.ssh\id_ed25519.pub | Set-Clipboard

Содержимое файла id_ed25519.pub скопировано в буфер обмена.

2.3 Добавить ключ в админке Selectel

Новый SSH-ключ
Имя
например: my-laptop
Публичный ключ
вставить содержимое id_ed25519.pub (начинается с ssh-ed25519 ...)
Сгенерировать пару SSH-ключей можно через терминал — используйте команду:
ssh-keygen -t ed25519

В панели Selectel: раздел SSH-ключей → «Добавить ключ» → вставить публичный ключ → «Добавить». Это даёт доступ к серверу по ключу при заказе/привязке сервера.

3. Подключение к серверу из VS Code

3.1 Проверка SSH из обычного терминала

# подключаемся к серверу по SSH (порт по умолчанию 22)
ssh root@135.106.144.236

# если используется нестандартный порт, например 22222:
ssh root@135.106.144.236 -p 22222

3.2 Добавить сервер в SSH config

Файл: ~/.ssh/config (Windows: C:\Users\<User>\.ssh\config)

# алиас хоста — позволяет подключаться командой "ssh abia-gitlab"
Host abia-gitlab
    HostName 135.106.144.236       # IP-адрес сервера
    User root                      # пользователь для входа по SSH
    Port 22                        # порт SSH на сервере
    IdentityFile ~/.ssh/id_ed25519 # приватный ключ для авторизации

Проверка: ssh abia-gitlab

3.3 Подключиться из VS Code

F1 / Ctrl+Shift+P
→ Remote-SSH: Connect to Host...
→ abia-gitlab

После подключения открыть рабочую папку:

File → Open Folder → /srv/gitlab

4. Установить Docker на сервер (Ubuntu)

Все команды выполняются в терминале VS Code, уже подключённом к серверу.

# проверяем название и версию ОС сервера
cat /etc/os-release

Сервер — Ubuntu → используем официальный apt-репозиторий Docker для Ubuntu.

4.1–4.2 Зависимости и ключ Docker

# обновляем список пакетов apt
apt-get update
# ставим зависимости, нужные для подключения репозитория Docker по https
apt-get install -y ca-certificates curl gnupg

# создаём каталог для хранения GPG-ключей репозиториев
install -m 0755 -d /etc/apt/keyrings
# скачиваем официальный GPG-ключ Docker
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
# даём всем пользователям право читать ключ
chmod a+r /etc/apt/keyrings/docker.asc

4.3–4.4 Репозиторий и установка

# добавляем официальный репозиторий Docker для Ubuntu в список источников apt
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  > /etc/apt/sources.list.d/docker.list

# обновляем список пакетов с учётом нового репозитория
apt-get update
# устанавливаем Docker Engine, CLI, containerd и плагины buildx/compose
apt-get install -y \
  docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

4.5–4.6 Запуск и проверка

# включаем автозапуск Docker и сразу запускаем службу
systemctl enable --now docker

# проверяем установленную версию Docker
docker --version
# проверяем установленную версию Docker Compose
docker compose version
# тестовый запуск контейнера — подтверждает, что Docker работает корректно
docker run hello-world

5. Подготовить папку GitLab

# создаём папку для конфигурации GitLab
mkdir -p /srv/gitlab/config
# создаём папку для логов GitLab
mkdir -p /srv/gitlab/logs
# создаём папку для данных GitLab (репозитории, БД и т.д.)
mkdir -p /srv/gitlab/data
# выставляем права доступа на всю структуру папок
chmod -R 755 /srv/gitlab
# переходим в рабочую папку проекта
cd /srv/gitlab

# выводим текущий путь — убедиться, что мы в /srv/gitlab
pwd
# выводим содержимое папки и проверяем, что подпапки созданы
ls -la

Ожидаемо: /srv/gitlab с подпапками config/ data/ logs/

6. Создать .env

# переходим в папку проекта (обязательно перед запуском compose)
cd /srv/gitlab
# открываем/создаём файл .env в редакторе nano
nano .env
GITLAB_HOST=135.106.144.236   # адрес сервера, используется как external_url
GITLAB_HTTP_PORT=80           # порт, на котором GitLab отвечает по http
GITLAB_SSH_PORT=2222          # порт для git-операций по SSH
⚠️ Важно: запускать docker compose нужно именно из /srv/gitlab, иначе .env не подставится и GitLab получит битый конфиг: 
gitlab_rails['gitlab_shell_ssh_port'] =   # пусто!

7. Создать docker-compose.yml

services:
  gitlab:
    image: gitlab/gitlab-ee:latest       # образ GitLab Enterprise Edition
    container_name: gitlab               # имя контейнера для docker ps / docker logs
    restart: always                      # автоперезапуск при падении или ребуте сервера
    hostname: ${GITLAB_HOST}             # имя хоста внутри контейнера (из .env)
    ports:
      - "${GITLAB_HTTP_PORT}:80"         # проброс web-интерфейса наружу
      - "${GITLAB_SSH_PORT}:22"          # проброс SSH для git clone/push/pull
    volumes:
      - /srv/gitlab/config:/etc/gitlab       # конфигурация хранится на хосте
      - /srv/gitlab/logs:/var/log/gitlab     # логи хранятся на хосте
      - /srv/gitlab/data:/var/opt/gitlab     # данные (репозитории, БД) хранятся на хосте
    shm_size: "256m"                     # увеличенная shared memory, нужна GitLab для стабильности
    environment:
      GITLAB_OMNIBUS_CONFIG: |           # конфиг GitLab Omnibus, применяется при старте контейнера
        external_url "http://${GITLAB_HOST}"                       # адрес, по которому доступен GitLab
        letsencrypt['enable'] = false                               # отключаем автоматический HTTPS
        nginx['listen_https'] = false                               # nginx слушает http, не https
        nginx['redirect_http_to_https'] = false                     # не редиректим на https
        gitlab_rails['gitlab_shell_ssh_port'] = ${GITLAB_SSH_PORT}  # порт SSH для git-операций

Поднимает: Web → http://135.106.144.236, Git SSH → ssh://git@135.106.144.236:2222/... (пока без HTTPS и DNS)

8. Запустить GitLab

# переходим в папку проекта (обязательно для подстановки .env)
cd /srv/gitlab
# запускаем контейнер GitLab в фоновом режиме
docker compose up -d

# проверяем, что контейнер запущен и слушает нужные порты
docker ps

Ожидаемо: gitlab/gitlab-ee:latest, health: starting, порты 0.0.0.0:80→80, 0.0.0.0:2222→22. Первый запуск может занять несколько минут.

9. Смотреть логи GitLab

# смотрим логи контейнера в реальном времени
docker logs -f gitlab
# выводим последние 200 строк логов
docker logs gitlab --tail=200

# проверяем статус внутренних служб GitLab (nginx, postgresql, redis и т.д.)
docker exec -it gitlab gitlab-ctl status

# проверяем статус healthcheck контейнера
docker inspect --format='{{json .State.Health.Status}}' gitlab
startingещё запускается
healthyGitLab готов
unhealthyесть проблема — смотреть logs

10. Проверить доступность

# выполняется с сервера

# проверяем, что GitLab отвечает локально (на самом сервере)
curl -I http://127.0.0.1
# проверяем, что GitLab отвечает по внешнему IP
curl -I http://135.106.144.236

С локальной машины открыть в браузере: http://135.106.144.236

⚠️ Именно http, не https — в конфиге nginx['listen_https'] = false и letsencrypt['enable'] = false

11. Открыть firewall

# смотрим текущий статус firewall
ufw status

# открываем порт 80 (web-интерфейс GitLab)
ufw allow 80/tcp
# открываем порт 2222 (SSH для git-операций)
ufw allow 2222/tcp
# применяем изменения правил firewall
ufw reload
# проверяем, что порты добавлены в правила
ufw status

# проверяем, какие порты реально слушает сервер
ss -tulpn | grep -E ':80|:2222|:443'

Ожидаемо в выводе: 0.0.0.0:80 и 0.0.0.0:2222

12. Первый вход в GitLab

# выводим содержимое файла с начальным паролем root
docker exec -it gitlab cat /etc/gitlab/initial_root_password

Найти строку Password: ...

Открыть http://135.106.144.236 и войти:

  • Username: root
  • Password: из файла initial_root_password

Итог

VS Code + Remote-SSH + Docker позволяют развернуть и администрировать GitLab полностью удалённо, без локальной установки чего-либо, кроме самого редактора.

Готово: GitLab доступен по http, с собственным SSH-портом для git-операций.
